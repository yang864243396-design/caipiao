# 管理后台前端方案（梳理稿）

> **范围**：仅前端信息架构与功能清单；**不涉及**后端接口设计或实现。  
> **依据**：仓库内 `client/src` 现有路由与页面能力（以 `router/index.ts` 为准）。  
> **现状**：仓库根目录已存在 `admin/` 脚手架（Vue 3 + TS + Vite，见 `admin/README.md`），本稿视为在其上扩展的产品蓝图。

> **待确认分流**：**原型 / Mock 前端方案**题干 **已定案至终**（**§8.8–§8.11** + **附录**）；**尚存**正式上线备忘 **§16**（**暂缓**）。**彩种目录** 主题：**§23～§29 已定案**（§23.11～§29.11）。**管理端扩展 IA**（**全站会员用户体系**、**用户侧方案查看/运营**）：**已定案** **§30.11**（Q143～Q148）。

## 2. 管理后台信息架构（菜单草案）

以下为**逻辑分组**，用于讨论与排期；实际菜单可在实现时合并或分拆。

### 2.1 总览与运维

| 模块 | 典型页面 | 说明 |
|------|----------|------|
| 工作台/仪表盘 KPI | 仪表盘 | 今日充值/提现、投注量、在线、告警摘要（展示结构先行，数据接 API 时填充） |
| 系统维护 | 维护开关、公告弹窗 | 全局维护页、强制刷新提示等 |
| 操作审计 | 审计日志列表 | 管理员关键操作留痕（标准能力） |

### 2.2 站点与内容（直接驱动 Client 展示）

| 模块 | 典型页面 | 与 Client 的对应关系 |
|------|----------|---------------------|
| 品牌与全局展示 | 站点信息、主题占位 | 品牌名、Logo、部分文案；与 `demoAppBrand` 等演示数据解耦后的正式配置入口 |
| 大厅运营 | 大厅装修 | Hero/Bento/最新动态数据源、入口排序与显隐 |
| 公告管理 | 公告 CRUD、发布状态 | `member-announcements`、`announcement-detail` |
| FAQ / 帮助 | 分类、文章、排序 | `member-faq`、`member-faq-detail`、`member-help` |
| 系统讯息模板 | 模板/广播 | `member-chat-system`；**路由与入口全员可见**（**§8.7 Q44**）；投递/分群 **上线前再定**（**§16 备忘**） |

### 2.3 游戏与玩法

| 模块 | 典型页面 | 与 Client 的对应关系 |
|------|----------|---------------------|
| 彩种/游戏目录 | 列表、上下架、排序、**稳定 code ↔ 展示名** | `play-detail`、自创方案、`copy-hall`、统计/报表等凡带「彩种」筛选处；**条目级说明见 §22** |
| 玩法与规则说明 | 规则文案、帮助链接 | 游戏详情页内容区块 |
| 跟单大厅运营 | 展示规则、推荐位（若有） | `copy-hall` |

> **实现状态（2026）**：`admin/` 已提供 **「游戏与玩法」** 下 **彩种目录、跟单大厅运营、倍投与方案模板（全局）、云端策略与监控** 等 Mock 页；**玩法与规则说明（独立 CMS）** 仍为 **未实现**（与首期定案一致）。

### 2.4 方案、倍投与云端策略

| 模块 | 典型页面 | 与 Client 的对应关系 |
|------|----------|---------------------|
| 倍投/方案默认策略 | 默认参数、风控阈值 | `bet-multiplier-settings` |
| 方案模板与高级参数 | 模板库、可下发字段 | `advanced-scheme-edit`、`advanced-scheme-rounds` |

### 2.5 会员与代理体系

| 模块 | 典型页面 | 与 Client 的对应关系 |
|------|----------|---------------------|

|------|----------|---------------------|

### 2.9 系统与安全（标准后台）

| 模块 | 典型页面 |
|------|----------|
| 管理员与登录 | 管理员账号；鉴权 **Bearer Access Token**（**§8.2 Q17**） |
| **角色管理** | 角色 CRUD、为角色分配 **菜单权限**；用户绑定角色（**§8.2 Q23**）；**超管角色不可删、至少一超管用户**（**§8.3 Q29**） |
| 角色与权限 | 运行时 **菜单级** 过滤（首版不做按钮级） |
| 审批中心 | **提现审批** 菜单：待办/已办、通过→**待打款**/驳回（**§8.3 Q27**）；**不含**「已打款」确认（该动作在 **提现出款** 菜单，**§8.4 Q33**） |
| 参数与字典 | 全局参数、枚举（**无多语言**，**§8**） |

## 3. Client 功能 → 后台「可配置项」梳理（推断）

下表将 **Client 已呈现能力** 映射为后台可能需要维护的配置或数据（实现时以实际业务为准）。

| Client 能力 | 后台推断职责 |
|-------------|--------------|
| 大厅品牌、Hero、Bento、动态 | 站点品牌资源；运营位内容；动态/公告条目或数据源 |
| 游戏详情 | 彩种元数据、图文、规则、状态（**详情与目录对齐见 §22**） |
| 跟单大厅 | 列表策略、筛选默认、运营配置 |
| 自创方案/方案配置/方案模式 | 方案模板、限制、周期/期次规则、默认可选项；**按用户实例查看/运营** 见 **§30** |
| 倍投设定 | 倍数方案、限额、允许档位 |
| 云端中心、回头 | 默认策略、阈值说明、开关（演示中为表单状态，后台需定是否全员默认或按用户）；**按会员查看/管理运行中方案** 见 **§30** |
| 投注记录（多入口） | 查询维度、保留周期、导出（后台统一数据服务） |
| 会员中心矩阵 | 功能显隐、角标（如 NEW）；**无多语言**（**§8**） |
| 充值/提现 | 渠道、银行/USDT 网络、费率、维护、单笔限额 |
| 银行卡 | 绑定上限、审核规则 |
| 资料/密码/昵称 | 策略：修改频率、强制实名等（若业务需要） |
| **全站会员/用户管理体系** | **§30.11-Q143～Q144**：最小闭环 + **只读/改上级/重置密码/冻结** 等均允许，由 **菜单级 RBAC** 分角色 | 不等同于 **§2.9**「管理员与登录」 |
| 公告/FAQ/帮助 | 内容 CMS 化 |
| 团队/推广 | **一级须后台挂靠**；**二级由一级在会员端配置**（**§8.5 Q38**）；**固定两级**（**§8.5 Q41**）；非团队功能与是否代理 **无关**（**§8.5 Q39**） |
| 报表类 | 统计口径、时区、数据延迟说明 |
| 聊天/系统讯息 | **原型**：后台/Mock 可用 **单一假列表**（**§8.8**）；**上线**：广播/分群见 **§16 Q48（暂缓备忘）**；**聊天** **全员**假会话 Mock + 置顶官方客服（**§8.4 Q35**、**§8.5 Q42**、**§8.6 Q43**）；**系统讯息页** **全员**可进（**§8.7 Q44**） |

## 4. 前端技术方案（草案，不改代码）

| 议题 | 建议 | 说明 |
|------|------|------|
| 工程位置 | 沿用 `admin/` 独立应用 | 与 `client` 发布隔离，权限边界清晰 |
| 技术栈 | Vue 3 + TypeScript + Vite + Element Plus | 与 `client` 一致，降低心智负担 |
| 路由结构 | 布局壳（侧栏 + 顶栏）+ 子路由 | 按第 2 节分组 Lazy load |
| API | **首版前端全量 Mock** | **§8.5 Q40**：可先 **不按权限** 展示全部页面与数据；上线后 **admin** 按 **§8.2 Q23**，**会员端** 按 **§8.5 Q39** + **§8.6** + **§8.7**（团队边界、系统讯息可见性）收敛 |
| 鉴权 | 登录页 + 路由守卫 + **Bearer Access Token** Mock | **无 Refresh**；过期 **统一提示并回登录页**（**§8.2 Q17** + **§8.3 Q28**）；**演示账号** `admin` / `admin123`（**§8.9 Q66**） |
| 审批 | **仅提现审核**（**§8.2 Q16** + **§8.3 Q32**）；状态机 **§8.3 Q27** | **出款确认** 独立菜单（**§8.4 Q33**） |
| 富文本 | **本地上传 + URL 外链**（**§8.2 Q20**）；**admin 选型独立**（**§8.8 Q55**） | **存 HTML**；**会员端与后台语义一致**（**§8.9 Q63**）；**原型不 sanitize**（**§8.10 Q71**）；**图片 ≤5MB**，**其它 ≤50MB**（**§8.3 Q30**）；**MIME 暂不限制**（**§8.4 Q36**） |
| 表格 Mock | **每页 10**、造数 **50**、**全站列表默认**（**§8.8 Q54** + **§8.9 Q65**） | **无** 规格化空/单条/超长样例要求 |
| 列表导出 | **Mock 无导出按钮**（**§8.8 Q59**） | 接 API 后再议 |
| 假数据主键 | 与 **`client` Mock 同字符串**（**§8.8 Q60**）；**维护**（**附录** + **§8.9 Q64**） | 双端联演示 |
| 报表 | 各报表页 **口径说明** | 与会员端 **同源**（**§8.2 Q21**） |
| 视觉 | **DESIGN.md 主色 + 字体** + 高密度表格 | 无暗色主题 |
| 与 Client 共享 | 暂不强制 monorepo 拆包 | **§8**：短期内不考虑拆仓库 |

### 「Q131」分类用来做甚么——白话框（产品与研发共读）

> **执行口径（首期）**：**无「分类」字段**（**§28.11-Q137**）。下列条目用于统一研发团队对 **「若未来启用分类」** 的语义，避免与开奖类型、供应商混为一谈。

- **为谁分类**：为 **每一条彩种目录记录**（`code`）分类，例如打个标签「分分彩 / 福彩系 / 低频」——**不是你说的「开奖结果类型」的分类**。**供应商/开奖源** 与「运营分类」**不是一回事**：**首期** 目录 **不设**「第三方归属 / 开奖源」**独立列**（**§28.11-Q138**），对账与第三方标识走 **服务端/运维文档**。  
- **用来干吗**：主要在 **后台** 做筛选、导出与权限类扩展；**可选用**来做 **前台** 大厅分组（你已允许能力开关全开，≠ 必须首期做前台分组）。  

### 开奖归因（连结 Q133）

- **平台不自算开奖号**：开奖 **有效号码/结果判定**依赖 **上游第三方**拉取或对账。**「跑出结果」** 指 **接收到合法第三方开奖事件并完成自身结算链路**，而非本机 RNG。  
- **风险**：第三方 **延迟 / 空缺 / 断连** 时：一般性 **软停 / 等待官方开奖** 见 **§27.11-Q133**；若属 **长期无开奖、断档**（**§28.11-Q139**）→ **§29.11-Q142 (B)**。
